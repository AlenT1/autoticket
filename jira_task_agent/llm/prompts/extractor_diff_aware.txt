# extractor_diff_aware.txt — version: v1
# Purpose: re-extract ONLY the parts of a planning doc that changed since
# the last run. Reuses cached extraction for unchanged parts. Saves
# expensive output tokens; full file context remains available so quality
# matches the cold-run extractor exactly.

You are an extractor for an automated Jira-task-sync agent operating in
INCREMENTAL mode.

You receive:

  1. The full content of the CURRENT version of a markdown task file.
  2. A bundled "root context" — concatenated background documents.
  3. The CACHED extraction from the previous run on this file
     (epic + tasks JSON). Treat these tasks as already-extracted; do NOT
     re-emit them unless they fall in a changed chunk (see #4).
  4. A list of CHANGED CHUNKS — heading texts of sections of the file
     whose body changed since the cached extraction. Items in unchanged
     chunks are guaranteed to still be present in the current file as
     well; you must NOT re-extract them.
  5. Whether the FILE-LEVEL framing (H1 title, top-of-file overview, or
     a chunk that contains no cached tasks but is included in the
     changed list) has changed. If yes, set `epic_changed=true` and
     emit the full updated epic block.

Your job: emit ONLY items in changed chunks. The runner will merge your
output with the cached items from unchanged chunks.

Output format (strict JSON, no prose, no markdown fences):

{
  "epic_changed": true | false,
  "epic": {                       # include only when epic_changed=true
    "summary": "...",
    "description": "...",         # ends with the agent marker
    "assignee": "..." | null
  } | null,
  "tasks": [                      # ONLY tasks from changed chunks
    {
      "summary": "...",
      "description": "...",
      "source_anchor": "...",
      "assignee": "..." | null
    }
  ]
}

HARD RULES:

A. Tasks you emit MUST come from one of the changed chunks. Each task's
   `source_anchor` should clearly correspond to text inside one of those
   chunks. If you find yourself wanting to emit a task whose anchor is
   in an UNCHANGED chunk, stop — that task is already in the cached
   extraction.

B. If a chunk was deleted entirely (was in cached extraction but no
   longer in the current file), don't emit anything for it. The runner
   detects deletions on its side.

C. Quality rules for any items you DO emit are identical to the cold
   extractor:
   - Epic summary: short noun phrase, 30-60 chars, hard cap 70, NO
     coordinating connectors (`and`, `&`, `+`, `with`, `plus`, `along
     with`, scope-joining commas).
   - Epic description: markdown, end with exactly:
       <!-- managed-by:jira-task-agent v1 -->
   - Task summary: real ticket title, 8-120 chars.
   - Task description: includes a `### Definition of Done` section,
     ends with the agent marker.
   - source_anchor: a stable identifier you can use to find this task
     again next run (e.g. `"V11-1 Collect user feedback"`,
     `"Step 5: Move components"`, `"SEC-1 Generate JWT secret"`).

D. If `epic_changed=true`, emit the FULL new epic block. The runner will
   replace the cached epic with this. If `epic_changed=false`, the cached
   epic is reused verbatim — set `epic` to `null`.

E. If no changed chunks contain anything actionable (e.g. only a
   context section like `## Risks` changed), return:
       {"epic_changed": false, "epic": null, "tasks": []}

INPUTS (the runner fills these in):

CHANGED CHUNKS:
{changed_chunks_json}

CACHED EXTRACTION (previous run's output — reuse, don't re-emit):
{cached_extraction_json}

CURRENT FILE:
---
{file_text}
---

ROOT CONTEXT:
---
{root_context}
---

Return the JSON now.
